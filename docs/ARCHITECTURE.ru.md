# Архитектура

## Общая схема

```
микрофон ─┐                       ┌─ живая стенограмма (абзацы по голосам)
          ├─ AudioHub ─ STT ─ демон ┼─ контуры: ⚡ ответы · тезисы · минутки
BlackHole ─┘   (чанки 3с)         │   · дежавю · имена · разметка реплик
                                  └─ NDJSON stdout ←→ stdin-команды (UI)

Стоп → пересборка записи → обновление графа → архив + Саммари → [Claude-разбор]
```

Всё — процессы на вашей машине. Сетевые вызовы только на localhost
(Ollama); облачный слой — отдельные, выключенные по умолчанию опции.

## Демон (src/daemon.py)

Один процесс, потоки-контуры вокруг общей стенограммы (`Transcript` с
локом): STT-цикл, мгновенные ответы, автотезисы и подсказки, живые
минутки, дежавю по Ядрам, опознание имён, разметка реплик, облачный
контур, heartbeat. События уходят в stdout построчным JSON
(`{"type": "transcript"|"thesis"|"hint"|…}`), команды приходят в stdin
(`hint`, `ask <вопрос>`, `summary`, `stop`). Любой UI подключается
поверх этого протокола; second-instance защищён flock.

## Диаризация: два прохода

1. **Живой**: эмбеддинг каждого чанка (ERes2Net, 512-dim) → трекер голосов
   с гистерезисом (порог 0.45, «серая зона», относительное правило смены,
   новый голос — по двум согласным чанкам). Эмбеддинги живут только в RAM.
2. **Оффлайн после стопа** (src/rebuild_transcript.py): полная запись
   диаризуется по каналам; эхо динамиков в микрофоне режется по перекрытию,
   голоса короче 10 с вливаются в соседей, сегменты заново транскрибируются,
   имена расставляет LLM (владелец = самый долгий голос своего микрофона).
   Живая версия сохраняется рядом как черновик.

## Пайплайн после встречи (src/graph_updater.py)

1. LLM извлекает из стенограммы JSON: название (2-3 слова), участники,
   темы, решения, поручения, сущности, Ядра.
2. Обновление графа: заметка встречи со связями `[[Папка/Имя|Имя]]`,
   upsert узлов Люди/Системы (датированные факты, история не стирается),
   Ядра — «Статус» перезаписывается, «Хроника» копится.
3. Архив (src/meeting_archive.py): папка «дата — название», человеческие
   имена файлов, «Вопросы и ответы» собираются из лога подсказок,
   Саммари генерируется с историческим контекстом (Ядра + два прошлых
   саммари; будущее не утекает в прошлое — отсечка по дате).
4. Опционально: облачный Claude сверяет минутки со стенограммой и
   дообогащает граф связями, которые видны только из истории.

## Граф знаний (папка Obsidian)

```
<graph_dir>/
  Встречи/YYYY-MM-DD_HHMM_название.md   ← эпизоды (сырьё не теряется)
  Люди/  Системы/  Команды/             ← сущности с обратными ссылками
  Ядра/                                 ← сквозные темы: Статус + Хроника
  Заметки/                              ← голосовые заметки
  Встречи-архив/дата — название/        ← слой для чтения (Finder)
  _MOC.md                               ← оглавление
```

Это трёхслойная схема «эпизоды → сущности → сообщества» (как в
Graphiti/Zep), но на чистом markdown: работает grep, Obsidian, git и
любой редактор. Устаревшие факты не удаляются — помечаются датой
вытеснения.

## Почему такие модели

Замеры и внешние источники — [MODELS.ru.md](MODELS.ru.md). Ключевое:
основная модель не ниже 30B-класса (планка graph-экстракции), лёгкая —
параллельно в RAM, `num_ctx` всегда задан явно.
